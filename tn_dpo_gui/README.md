# TN-DPO GUI Subproject

This folder is an isolated Python subproject for personalized GUI-agent Task-Nullspace DPO training. It lives inside your `GUI2` repository, but it can now reuse the main project's real data/model/output directory conventions from the root `config.yaml`.

## What is included

- A normalized GUI-step / trajectory data schema.
- Candidate generation from logged actions, user history, UI affordances, and safe negatives.
- Leakage-aware history retrieval and continuation retrieval.
- State-level task-nullspace projection:
  `rho = sum(w * m_pref * m_task) / (sum(w * m_task^2) + eps)`
- Pair weighting:
  `omega = sigmoid((abs(m_null) - lambda_u * U) / tau_omega)`
- Weighted DPO ranker training.
- Capacity-regression gate training.
- Offline evaluation for ranker, gate, and projection quality.
- A deterministic demo pipeline with smoke-testable sample data.

## Project layout

- `tn_dpo_gui/`: Python package.
- `configs/`: local configs used only by this subproject.
- `tests/`: focused unit tests.
- `artifacts/`: demo smoke outputs.
- `data/demo/`: demo smoke inputs generated by the preprocessing script.

## Quickstart

Install dependencies from this folder:

```bash
pip install -r requirements.txt
```

Run the demo smoke pipeline on Linux/macOS:

```bash
./run_pipeline.sh
```

Run the demo smoke pipeline on Windows PowerShell:

```powershell
.\run_pipeline.ps1
```

Or run each stage manually:

```bash
python -m tn_dpo_gui.scripts.preprocess_data --demo --output-dir data/demo
python -m tn_dpo_gui.scripts.build_user_index --trajectories data/demo/trajectories.jsonl --output data/demo/user_index.json
python -m tn_dpo_gui.scripts.build_pairs --config configs/build_pairs.yaml
python -m tn_dpo_gui.scripts.train_ranker --config configs/train_ranker.yaml
python -m tn_dpo_gui.scripts.train_gate --config configs/train_gate.yaml
python -m tn_dpo_gui.scripts.eval --config configs/eval.yaml
```

## Main-project mode

For real training on the same data layout used by the main PAPO pipeline, first prepare the main project data as usual, then run:

```bash
./run_main_project_pipeline.sh
```

or on Windows:

```powershell
.\run_main_project_pipeline.ps1
```

This mode reads the root [`config.yaml`](../config.yaml) and derives TN-DPO paths from the main project layout:

- processed TN-DPO data: `paths.work_dir/tn_dpo_gui/processed/`
- pair artifacts: `paths.work_dir/tn_dpo_gui/pairs/`
- ranker checkpoints: `paths.checkpoint_root/tn_dpo_gui/ranker/`
- gate checkpoints: `paths.checkpoint_root/tn_dpo_gui/gate/`
- evaluation report: `paths.logging_root/tn_dpo_gui/report.json`

The adapter consumes these existing main-project artifacts:

- `paths.task_dir/execution_train_config.jsonl`
- `paths.task_dir/execution_eval_config.jsonl`
- `paths.work_dir/papo_steps.jsonl`

If they are missing, run the main PAPO preparation flow first, such as:

```bash
python scripts/14_build_data_protocol.py --config config.yaml
python scripts/09_run_config_pipeline.py --config config.yaml
```

## Design notes

- The default encoder is deterministic hashing so the project runs without downloading external embedding models.
- If you later want stronger text features, set `encoder.backend: sentence_transformers` and install `sentence-transformers`.
- Relative paths inside YAML configs are resolved from the config file's own directory, not from the shell's current working directory.
- Ranker and gate training now read only `train` pairs by default, while offline evaluation reads only `eval` pairs by default.
- History retrieval never reads from `test` split data and excludes the current example's own trajectory from user-history lookup.
- Continuation retrieval excludes the current trajectory from offline retrieval and only reuses the current example's future trajectory for `train/history` samples when the queried action matches the logged action.
- The gate defaults to the attachment-aligned `target_mode: net_capacity`, so the regression target is `capacity - gate_cost`.
- In main-project mode, missing PAPO step coverage for any target episode is treated as an error instead of silently backfilling from evaluation-only `target.actions`.
- Build/train/eval artifacts now include `resolved_config.json` plus provenance fields, and JSON / JSONL / checkpoint writes use atomic replace semantics.
- In main-project mode, the training checkpoints also record the root config's `model_name_or_path` for experiment provenance.
